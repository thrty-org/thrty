# Composition order

The order you list middlewares inside `compose(...)` decides what each middleware can see and catch. `compose(A, B, C)(handler)` produces a function that, when called, executes:

1. **Pre-phase** вАФ top-down: `A`'s code that runs *before* it calls `next` вЖТ `B`'s pre-code вЖТ `C`'s pre-code вЖТ the actual `handler`.
2. **Post-phase** вАФ bottom-up: the actual handler returns вЖТ `C`'s post-code (after `await next(...)`) вЖТ `B`'s post-code вЖТ `A`'s post-code.

This rule governs every other rule on the page.

If `C` throws, the error escapes upward вАФ `B` sees it, then `A` sees it. If `A` has a `try { return await next(event, context); } catch { вА¶ }`, it catches the throw. If `C` itself is the try/catch, it cannot catch *its own* upstream вАФ only what's below it.

## The catch-only-downstream rule

An `errorHandler` only catches throws from middlewares registered *after* it. This makes placement matter вАФ always place `errorHandler` above the middlewares it should catch.

```typescript
// вЬЕ errorHandler sits above validate вЖТ catches its throws
compose(
  typesOf<Handler<unknown>>(),
  errorHandler({ onError }),
  validate(schema),
);

// вЭМ errorHandler sits below validate вЖТ throws escape past it
compose(
  typesOf<Handler<unknown>>(),
  validate(schema),
  errorHandler({ onError }),
);
```

The `вЭМ` example *looks* fine because `errorHandler` is listed second. But `validate` runs first in the pre-phase, and its throw bubbles upward (toward `typesOf`), not downward (toward `errorHandler`). The `errorHandler` is below it in the chain вАФ i.e. closer to the actual handler вАФ and never sees the throw at all.

## Middlewares that catch internally

Some middlewares wrap downstream calls with their own `try/catch`. `@thrty/sqs-record-iterator` with `batchItemFailures: true` is one вАФ it catches per-record throws and converts them into a `SQSBatchResponse`. An `errorHandler` placed above the iterator will only see throws from the iterator itself (rare), not the per-record errors.

```typescript
compose(
  typesOf<SQSHandler>(),
  errorHandler({ onError }),                 // вЖР never sees per-record throws
  forEachSqsRecord({ batchItemFailures: true }),
  validate(schema),                          // throws here for one record
);
```

When `validate` throws, `forEachSqsRecord`'s internal catch handles it; the outer `errorHandler` only sees `Promise.resolve(SQSBatchResponse)`.

To intercept per-record errors before the iterator catches them, place `errorHandler` between the iterator and the throwing middleware:

```typescript
compose(
  typesOf<SQSHandler>(),
  forEachSqsRecord({ batchItemFailures: true }),
  errorHandler({                             // вЖР inside the iterator
    onError: (e) =>
      e instanceof ValidationError ? undefined : Promise.reject(e),
  }),
  validate(schema),
);
```

See [Use cases вЖТ SQS consumer](/use-cases/sqs-consumer) for the full pattern.

## The general rule

For any middleware that wraps with `try/catch` (or `.catch`, or `next().catch(вА¶)`):

- It can only see throws from middlewares **listed after it** in `compose`.
- A middleware that catches its own internal errors (the iterator's `batchItemFailures`, retry loops, breakers) hides those errors from anything above it.

When you debug an "unhandled error" you *thought* was caught, walk the chain in `compose` order and identify where the throw originates vs. where the catcher sits. If the catcher is upstream of the throw, move it.
